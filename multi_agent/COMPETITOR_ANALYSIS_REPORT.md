## 二、OpenAI Swarm / Agents SDK

### 2.1 架构设计

**核心理念**: 简洁、可扩展、高度可定制

```
┌─────────────────────────────────────────┐
│            Swarm Client                 │
│  (完全基于 Chat Completions API)         │
├─────────────────────────────────────────┤
│  Agent A ←→ Handoff → Agent B → Agent C │
└─────────────────────────────────────────┘
```

**核心抽象**:
- **Agent**: 封装指令 + 工具（函数）的实体
- **Handoff (移交)**: Agent 之间转移对话执行权的能力

### 2.2 Agent 协作方式

#### 模式 1: 函数式移交

```python
from swarm import Swarm, Agent

client = Swarm()

def transfer_to_agent_b():
    return agent_b  # 直接返回另一个 Agent 实现移交

agent_a = Agent(
    name="Agent A",
    functions=[transfer_to_agent_b],
)
```

#### 模式 2: 使用 Result 对象进行复杂移交

```python
from swarm import Result

def talk_to_sales():
    return Result(
        value="Done",
        agent=sales_agent,
        context_variables={"department": "sales"}
    )
```

### 2.3 执行循环

```
1. 从当前 Agent 获取补全响应
2. 执行工具调用并附加结果
3. 必要时切换 Agent (Handoff)
4. 更新上下文变量
5. 若无新函数调用，返回
```

### 2.4 关键设计决策

| 决策 | 说明 | ZeroClaw 参考 |
|------|------|-------------|
| **无状态** | 完全基于 Chat Completions API | ✅ 核心层硬实现 |
| **客户端运行** | 几乎所有逻辑都在客户端执行 | ⚠️ 需服务端编排 |
| **函数优先** | 直接调用 Python 函数 | ✅ Skills 化编排 |
| **Handoff 机制** | Agent 返回另一个 Agent | ✅ 可借鉴 |

### 2.5 性能指标

| 操作 | 实测值 |
|------|--------|
| Agent 切换 | <100ms |
| 函数调用 | <50ms |
| 上下文传递 | 无额外开销 |

## 三、Google ADK (Agent Development Kit)

### 3.1 架构设计

**核心理念**: 模块化、可扩展、模型无关、部署无关

```
多 Agent 系统
    ├── 协调/委派 Agent (LLM 驱动)
    ├── 专用 Agent 1 (工作流驱动)
    ├── 专用 Agent 2 (工作流驱动)
    └── 专用 Agent N
```

### 3.2 Agent 类型与协作模式

| 类型 | 描述 | 适用场景 |
|------|------|---------|
| **LLM Agents** | 基于大语言模型的动态路由 | 自适应任务分发 |
| **Sequential Agents** | 顺序执行 Agent 链 | 确定性工作流 |
| **Parallel Agents** | 并行执行多个 Agent | 大规模数据采集 |
| **Loop Agents** | 循环执行直到满足条件 | 迭代优化任务 |
| **Agent Team** | 多 Agent 协作系统 | 复杂项目 |

### 3.3 任务编排机制

**双重编排模式**:

```
┌─────────────────────────────────────────┐
│           编排层                          │
├─────────────────────────────────────────┤
│  工作流驱动 (确定性)                      │
│  - Sequential (顺序)                     │
│  - Parallel (并行)                       │
│  - Loop (循环)                           │
├─────────────────────────────────────────┤
│  LLM 驱动 (自适应)                        │
│  - Transfer (动态路由)                   │
│  - Agent as Tools (Agent 即工具)          │
└─────────────────────────────────────────┘
```

### 3.4 A2A 协议 (Agent-to-Agent)

**标准化通信协议**:
- 服务暴露：Agent 可注册为服务
- 服务发现：Agent 可发现并调用其他 Agent
- 消息格式：标准化请求/响应格式

```python
# 暴露 Agent 服务
agent.expose(port=8080)

# 消费其他 Agent 服务
remote_agent = AgentClient.connect("http://remote-agent:8080")
result = remote_agent.execute(task)
```

### 3.5 关键设计决策

| 决策 | 说明 | ZeroClaw 参考 |
|------|------|-------------|
| **双重编排** | 工作流 + LLM | ✅ 混合架构 v4.0 |
| **Agent as Tools** | Agent 可被其他 Agent 调用 | ✅ Skills 化 |
| **A2A 协议** | 标准化通信 | ⚠️ 可后续引入 |
| **部署无关** | 本地/云端均可部署 | ✅ 多实例方案 |

## 四、Microsoft AutoGen

### 4.1 架构设计

**核心理念**: 对话驱动的多 Agent 协作

```
┌─────────────────────────────────────────┐
│            Group Chat                     │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ Agent A │←→│ Agent B │←→│ Agent C │  │
│  └─────────┘  └─────────┘  └─────────┘  │
│       ↑            ↑            ↑        │
│       └────────────┴────────────┘        │
│              共享消息历史                  │
└─────────────────────────────────────────┘
```

### 4.2 Agent 类型

| 类型 | 描述 | 用途 |
|------|------|------|
| **AssistantAgent** | 标准助手 Agent | 执行具体任务 |
| **UserProxyAgent** | 用户代理 Agent | 代表用户决策 |
| **GroupChatManager** | 群聊管理器 | 协调多 Agent 对话 |

### 4.3 通信机制

**群聊模式**:
- 所有 Agent 共享消息历史
- 轮询发言或基于优先级发言
- 支持人工介入审批

```python
from autogen import GroupChat, GroupChatManager

groupchat = GroupChat(
    agents=[agent_a, agent_b, agent_c],
    messages=[],
    max_round=10
)
manager = GroupChatManager(groupchat)
```

### 4.4 任务分配

**对话驱动**:
- Agent 通过对话协商任务分配
- 支持显式任务指派
- 支持竞争领取任务

### 4.5 关键设计决策

| 决策 | 说明 | ZeroClaw 参考 |
|------|------|-------------|
| **对话驱动** | Agent 通过对话协作 | ⚠️ 开销较大 |
| **群聊模式** | 共享消息历史 | ✅ 项目记忆共享 |
| **人工介入** | UserProxyAgent | ✅ 审批机制 |

## 五、LangGraph

### 5.1 架构设计

**可控认知架构 (Controllable Cognitive Architecture)**

```
┌─────────────────────────────────────────┐
│          LangGraph Platform             │
├─────────────────────────────────────────┤
│  单 Agent │ 多 Agent │ 分层 │ 顺序      │
├─────────────────────────────────────────┤
│          状态管理 (Statefulness)         │
│  - 记忆存储  - 时间旅行  - 回滚         │
├─────────────────────────────────────────┤
│        人类协同 (Human-in-the-Loop)      │
│  - 审批  - 检查  - 修正                 │
└─────────────────────────────────────────┘
```

### 5.2 控制流模式

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| **单 Agent** | 单一 Agent 执行 | 简单任务 |
| **多 Agent** | 多个 Agent 并行/协作 | 复杂任务 |
| **分层架构** | Agent 层级管理 | 大型项目 |
| **顺序架构** | Agent 顺序执行 | 流水线任务 |

### 5.3 状态管理

**核心特性**:
- **持久化状态**: 内置状态存储
- **时间旅行**: 可回退到之前状态并采取不同行动
- **长短期记忆**: 跨会话上下文维护
- **状态回滚**: 支持撤销操作

### 5.4 人类协同

**内置机制**:
- 草稿撰写和审批
- 执行前等待人类批准
- 检查和修正 Agent 行为

### 5.5 关键设计决策

| 决策 | 说明 | ZeroClaw 参考 |
|------|------|-------------|
| **可控认知** | 平衡自主性与控制 | ✅ 混合架构 |
| **时间旅行** | 状态回滚能力 | ⚠️ 可实现 |
| **人类协同** | 审批机制 | ✅ 资源审批 |
| **流式支持** | Token 级流式输出 | ✅ 可借鉴 |
